Orbit Wars: IMPALA-lite Self-Play Design

Detailed design memo for impala_v1 and the planned impala_v2 runs. For teammates / reviewers. 2026-04-19.

TL;DR. A Set-Transformer policy with 0.9 M parameters, bootstrapped from a behaviour-cloning checkpoint, trained via synchronous on-policy actor-critic self-play (four parallel CPU workers). Reward is a decaying mix of Orbit-Wars-specific shaped rewards (planet count, production rate, sun-kill penalty, enemy kills) and sparse terminal ±1. Every seat in every game contributes to the gradient (winners and losers). v2 adds a 50 % probability that any non-learner seat plays with the rules-based lb-928 planner instead of the current policy, to avoid self-play echo chambers.

1. Problem

Orbit Wars is a 2-or-4-player RTS on a 100×100 continuous board with a sun at the centre. Each turn, every player simultaneously submits a list of moves, where each move is [source_planet_id, angle_radians, num_ships]. Planets produce ships each turn proportional to their radius; inner planets rotate, outer ones are static. Fleets travel in straight lines at a speed scaling with fleet size; crossing the sun (point-to-segment distance < 10) destroys them. Combat on a planet resolves as top-attacker minus second-attacker survivors, with ownership flipping if the garrison is overwhelmed. Game ends at step 500 or when only one player has planets or fleets remaining. Winner is argmax of total ships; reward is ±1 terminally, 0 elsewhere.

Concretely:

  • Perfect information, simultaneous moves, deterministic given seed.
  • Variable-cardinality state: ~20–40 planets plus 0–∞ fleets, changing across turns.
  • Multi-component action per turn: one sub-action per owned planet. Joint action space is exponential in the number of owned planets.
  • Hard-limit actTimeout = 1 s per turn (with 2 s bankable overage). Deployment can't run heavy search.

2. Model: OrbitAgent (0.9 M params, shared policy+value)

2.1 Observation encoding

Three input streams; each entity becomes a 128-d token.

Planet features (one row per planet; shape [N, 14])

[0] owner_is_me       one-hot for "I own this planet"
[1] owner_is_enemy    one-hot for "another player owns this"
[2] owner_is_neutral  one-hot for "owner == -1"
[3] x_norm = (x - 50) / 50              board coordinate normalised to [-1, 1]
[4] y_norm = (y - 50) / 50
[5] radius                               radius (float)
[6] log_ships = log1p(ships) / 8         compressed ship count
[7..11] prod_onehot_1..5                 production rate one-hot ∈ [1, 5]
[12] is_static                           orbital_r + radius >= 50
[13] is_comet                            spawned after t=0 (temporary planet)

Fleet features (one row per active fleet; shape [F, 9])

[0] owner_is_me
[1] owner_is_enemy
[2] x_norm, [3] y_norm
[4] sin(angle), [5] cos(angle)
[6] log_ships
[7] from_planet_idx_norm                 source-planet id normalised
[8] eta_norm                             crude time-to-edge-of-board / 50

Global features (one vector; shape [16])

[0] step / 500
[1] angular_velocity (radians / turn)
[2] (500 - step) / 500                   remaining-turn fraction
[3..6] player_slot one-hot (0..3)
[7] is_4p
[8..10] n_my / n_enemy / n_neutral planets, each / 20
[11..12] log1p(my_total_ships)/8, log1p(enemy_total_ships)/8
[13] comet_window_active                 step within ±5 of scheduled spawn
[14..15] sin, cos(angular_velocity * step)   current rotation phase

2.2 Architecture

planets [N, 14]  →  MLP + type_embed(0)  →  [N, 128]  (planet tokens)
fleets  [F, 9]   →  MLP + type_embed(1)  →  [F, 128]  (fleet tokens)
globals [16]     →  MLP + type_embed(2)  →  [1, 128]  (global token)

        ↓  concatenate: [N + F + 1, 128]

        ┌──────────────────────────────────────────────┐
        │  4 × Set-Attention Block  (pre-LN MHA + FFN) │
        │  d_model = 128, n_heads = 4, ff_expansion = 4 │
        │  permutation-invariant over entity tokens     │
        └──────────────────────────────────────────────┘

        ↓  planet_tokens: [N, 128]   global_token: [128]

  Policy heads (per planet token):
    target_q = Linear(planet_token)
    target_k = Linear([pass_token, planet_tokens])        # [P+1, d]
    target_logits = q · kᵀ / √d                            # [P, P+1]
    bucket_logits = Linear(planet_token)                   # [P, 4]

  Value head (on global token):
    V(s) = MLP( global_token ) → ℝ

Total parameters ≈ 0.9 M. CPU forward on one sample is 3–5 ms; batch of ~100 samples is ~20–40 ms.

2.3 Action factorisation

An action at turn t is a set a_t = { (src_i, tgt_i, bucket_i) : i ∈ owned_planets_with_ships }. For each owned planet i, the policy independently samples:

  • tgt_class ∈ {pass, planet_0, …, planet_{P-1}}, where pass (class 0) means "this planet sits still this turn".
  • ships_bucket ∈ {25 %, 50 %, 75 %, 100 %} of the garrison.

Given these, the env move is recovered deterministically:

angle     = atan2(target.y - src.y, target.x - src.x)
num_ships = round(bucket_fraction × src.ships)
Why this factorisation? In our scraped top-10 data, every action is effectively a planet-directed shot — nobody aims into empty space. Reducing the continuous angle to "which planet are you attacking" collapses a continuous decision into a discrete one, saves model capacity, and aligns the action prior with actual expert behaviour.

Joint log-probability

Because sub-actions are sampled independently per planet, the joint log-probability of the full turn's action is the sum over owned planets:

log π(at | st) = Σi ∈ owned [ log πtgt(tgti | st, planeti) + log πbkt(bucketi | st, planeti) ]

This is what enters the policy-gradient term below.

2.4 Why one shared encoder (not two separate networks)

The name "Actor-Critic" can mislead into thinking we need two networks — one to act, one to evaluate. In practice, the actor / critic distinction is about roles, not about how they're implemented. Our OrbitAgent has one shared Set-Transformer encoder with three heads: two policy heads (target + bucket) and one value head. This is the standard choice in modern deep RL.

SystemArchitecture
AlphaGo / AlphaZero (Silver 2017) Shared ResNet tower, separate policy and value heads
Lux AI S1 winner (Pressman 2021) Shared ResNet backbone (24 blocks), three actor heads (worker/cart/citytile) + value head
CleanRL PPO (Huang 2022) Shared MLP encoder, two linear heads
A3C original (Mnih 2016) Paper recommends shared backbone in §3
Our OrbitAgent Shared Set-Transformer (4 layers, d=128) + target_q·target_k / bucket_linear / value_mlp

Why share

  • Sample efficiency. The state representation the value head needs — "is my side winning right now?" — is built from the same information the policy head needs — "where should I attack?". Sharing lets every gradient update on either head improve the shared representation for both.
  • Compute. One forward pass yields target, bucket, and value simultaneously. A two-network version doubles forward cost. For a Set-Transformer with 30+ tokens per sample at 3-5 ms CPU inference, doubling matters at the Kaggle actTimeout = 1s.
  • Parameter budget. 0.9 M shared params are enough for our task. Splitting into two 0.5 M independent networks loses the shared representation with no net parameter gain.
  • Inductive bias. In Orbit Wars the features that inform "where to attack" (planet ownership, positions, ship counts, rotation phase) are the same features that inform "am I winning"; no reason to learn two different representations of them.

When two networks would be preferred

Decoupled PPO (Cobbe, Hilton, Klimov, Schulman, 2021) shows that for some continuous-control tasks, the policy-loss gradient and the value-loss gradient can conflict in the shared encoder and training diverges; there they recommend a separate value network. This has not been reported for RTS / board-game domains, and our training signal stays well-behaved in our prior runs. If we later observe value-loss stuck high while policy-loss improves (or vice versa), splitting is the next thing to try.

Head summary

planets, fleets, globals → (embeddings + type_embed)
                         → [N + F + 1, 128]
                         → 4× SetAttentionBlock
                         → planet_tokens [N, 128], global_token [128]

  planet_tokens → target_q · target_k([pass_tok, planet_tokens])
                → target_logits [N, P+1]              # "where to attack per owned planet"
  planet_tokens → bucket_head linear
                → bucket_logits [N, 4]                 # "send how many ships"
  global_token  → value_head MLP
                → V(s) scalar                           # "am I winning right now?"

"A2C needs two networks" is an artefact of the original Barto-Sutton (1983) tabular formulation, repeated in some pedagogical materials. In any deep-RL implementation published after ~2015, the shared-backbone form is the standard.

2.5 Inference-time action masks

Only used at deployment, not during BC or RL training (masking at training would sometimes reject expert / self-sampled labels and blow up the cross-entropy).

  • Sun blocker: any (src, target) pair whose connecting line segment passes within 10 of the board centre has its target_logits entry set to −10⁹ before softmax.
  • Self-target: the diagonal is masked (can't shoot your own planet).

3. Starting Point: bc_v2

All RL runs bootstrap from bc_v2, a behaviour-cloning checkpoint trained on:

  • ~500 starter vs {starter, random} locally-generated games,
  • ~41 winner trajectories from 50 scraped top-10 Kaggle replays,
  • total ~230 k labelled (obs, action) pairs,
  • 30 epochs, best loss 0.099, local win rates: 95 % vs random, 20 % vs starter, 0 % vs lb-928.

The value head was trained only as an auxiliary regression target during BC (it predicts 0 early in the game, close to ±1 near terminal). We take both policy and value weights into the RL phase.

4. Reward Function (the key design choice)

The env supplies only a sparse terminal ±1. Training on that directly starves the early learner of signal. We add a dense shaped reward tuned to Orbit Wars, and decay its weight to zero over the first 50 iters so the policy eventually optimises the true objective.

4.1 Per-step shaped reward (per seat)

rshape(t, seat) =
     +0.10 · ΔP(seat)planet_count
     +0.02 · ΔP(seat)production_rate
     −0.10 · (ships_lost_to_sunthis_turn) / 100
     +0.05 · Δ(enemy_ships_destroyed) / 100
, clipped to [−1, +1]

Intuition behind each term:

  • +0.10 × Δplanets: territory is the most stable proxy for long-term ship production. Rewards capture, penalises loss. Largest coefficient.
  • +0.02 × Δproduction_rate: planets differ in production (1–5 ships/turn, encoded by radius). Capturing a 5-production planet is ~5× more valuable than a 1-production one, so we reward the rate differential directly on top of the planet-count term.
  • −0.10 × sun_kills / 100: ships lost to the sun are pure waste — worst possible move. Penalised harder than most gains are rewarded.
  • +0.05 × enemy_killed / 100: offensive pressure even if no planet changes hands. Rate-limited to keep this from dominating short-term greedy combat over long-term territory.

The ΔX notation is literally X_now − X_prev between consecutive turns of that seat's perspective.

4.2 Shape weight decay

To make sure we eventually optimise the true sparse ±1 objective:

wshape(iter) = max(0, 1 − iter / 50)

So:

  • Iter 0–10: shape weight 1.0, policy driven mostly by shaping.
  • Iter 11–49: shape weight linearly decreasing; terminal reward slowly takes over.
  • Iter 50–125: shape weight = 0; pure terminal ±1 — authentic objective.

4.3 Return (per recorded state)

For each (state, action) recorded for a seat during a game, the target return is:

G(t, seat) = wshape(iter) · Στ=t..T γτ−t · rshape(τ, seat) + terminal_reward(seat)

with γ = 0.997. This is a Monte Carlo (not TD-λ) return — simpler, and with 500-step episodes the variance is manageable.

Design comment. The terminal ±1 is always added, even during the shaped-reward phase, so the value head is never predicting a meaningless quantity. What changes with w_shape is how much weight the dense intermediate-reward component carries.

5. Loss Function

All four components computed in the same forward pass (since the model shares an encoder). Every entry in the batch can be from any seat — winner or loser.

5.1 Policy gradient

advantage(b) = G(b) − Vθ(sb) ← .detach()
advantage ← (advantage − mean) / (std + 1e-6) ← batch-normalised
advantage ← clamp(advantage, −2, 2) ← stability clip

Lpolicy = −Eb[ advantage(b) · log πθ(ab | sb) ]

Because the advantage is positive for above-value outcomes (mostly winners and not-so-losing losers) and negative for below-value outcomes, the policy gradient pushes log-probability up when we did better than the value head expected, and down when worse. Crucially this handles losers correctly: their advantage is negative, so −(−) · log π minimises toward lower log-probability of those losing actions.

5.2 Value regression

Lvalue = ½ · Eb[ ( Vθ(sb) − G(b) )² ]

5.3 Entropy bonus

H(s) = − Ec[ πtgt(c | s) · log πtgt(c | s) ]
Lentropy = −Eb[ H(sb) ] (i.e. subtract H to maximise entropy)

We only entropy-regularise the target head (not the bucket head) since the target head is the higher-stakes decision.

5.4 Total loss

L = Lpolicy + 0.5 · Lvalue − 0.01 · (−Lentropy)

Equivalently: loss = policy_loss + 0.5*value_loss - 0.01*entropy.

5.5 What we do NOT have (and why)

Not usedReason
Teacher KL (anchor to BC) BC is our starting point, not a strong upper bound; pulling the policy back to BC after it starts learning would cap progress. v1 sets teacher_kl_coef = 0 and the teacher forward pass is a no-op diagnostic.
PPO clipping ratio Our rollouts are synchronous and on-policy: each iter's rollouts are generated by the weights we just broadcast, then we update. Importance ratio ≈ 1; clipping is unnecessary noise at this batch size.
V-trace off-policy correction Same reason — nothing off-policy to correct for.
UPGO Would need reliable Q(s, a); we don't train a Q head. Advantage normalisation + clip is our cheap substitute.
TD(λ) returns Episodes cap at 500 steps and Monte-Carlo returns are the cleanest credit-assignment when episodes finish. If we add long self-play chains or horizon-extending rewards later, we'd add TD(λ).

6. Training Loop (impala_v1)

persistent Pool of 4 workers, each:
    imports torch once
    loads OrbitAgent (empty weights)
    disables env schema validation (~20% speedup)

for iter in 0 .. 124:
    broadcast current model state_dict bytes → 4 workers

    each worker independently:
        n_players ∼ uniform {2, 4}                  (four_player_prob = 0.5)
        create env, env.reset(num_agents = n_players)

        while not env.done:
            for seat in range(n_players):
                obs_shared = env.state[0].observation       # shared world state
                obs_for_seat = {**obs_shared, player: seat}
                action = model.forward(obs_for_seat)         # stochastic sample
                actions[seat] = action
            env.step(actions)                                # simultaneous apply

        featurise per-seat (obs, action) tuples at each acting turn
        compute per-step shaped reward trajectory
        compute discounted shape return
        attach terminal reward ±1 per seat
        return all records to master

    master gathers all records across 4 games:
        compute advantages (shape_weight · G_shape + terminal - V(s))
        normalise + clip
        gradient step × grad_steps_per_iter (default 2):
            L = policy_loss + 0.5 * value_loss - 0.01 * entropy
            backward, clip grad norm to 1.0
            AdamW step

    every 10 iters: torch.save( model + kwargs + iter, out_path )

Why synchronous, not asynchronous?

True IMPALA is actor-learner async with V-trace. For a CPU-only 4-worker setup, synchronous Pool.map() is simpler and removes the importance-sampling complexity. Actors generate, master trains, weights broadcast again — the "actor lag" is at most one iter rather than continuous.

7. Why Every Seat Contributes (both winners and losers)

A common pitfall we explicitly avoid: training only on the winner's trajectory. Arguments against that:

  • In 2-player games, half the data is the loser; in 4-player, three quarters. Throwing it away wastes sample efficiency.
  • Losers' moves are negative examples. Their terminal reward is −1, so their advantages are negative, and the policy gradient naturally pushes down log-probability of those choices. This is gradient-correct, just the sign is flipped relative to winners.
  • The value head especially benefits: only seeing winning states biases it optimistically; it needs losing states to learn that some configurations are genuinely bad.

Concretely in our implementation:

for seat in range(n_players):
    records[seat] = [...all (obs, action) at this seat's acting turns...]
    for feat in records[seat]:
        feat["_terminal_reward"] = rewards[seat]   # +1 if winner else -1
        feat["_shape_return"]   = monte_carlo_discounted_sum_for_seat

# In master:
all_samples = flatten(records over all workers, all seats)
advantage  = normalise_clip(return - V)
policy_loss, value_loss computed across ALL samples.

8. impala_v2: Add lb-928 Opponent Mixing

Pure self-play has a well-known failure mode: the agent learns to play against its own evolving distribution, not against real-world opponents. On the Kaggle leaderboard the biggest fixed opponent we'll face is the rules-based lb-928 planner (currently in the public top 10). impala_v2 adds it as a mix-in opponent during self-play.

8.1 Opponent assignment per game

for each game started by a worker:
    learner_seat = random.randint(0, n_players - 1)
    for seat in range(n_players):
        if seat == learner_seat:
            seat_policy[seat] = current_model         # always self
        else:
            if random() < 0.5:
                seat_policy[seat] = current_model     # self
            else:
                seat_policy[seat] = lb928_agent_fn    # fixed external

8.2 Training-data rule

Seat typeRecord data?Why
Learner (always self)✅ yes — policy + value loss It's our model acting; standard on-policy data.
Non-learner self✅ yes — policy + value loss Same policy, just a different seat.
Non-learner lb-928❌ skip entirely The action distribution is rules-based, not derived from πθ. Training on those actions would pull the policy toward imitating lb-928, which we tried via BC distillation (bc_v3) and found regressed performance. We want the lb-928 seats only as opposition, not as a training target.

The self seats' training gradients are still correct: their advantages use their own seat's terminal reward (+1 winner, −1 loser).

8.4 Expected effect

  • vs lb-928 eval win rate: should improve from 0 % (our current floor) toward something non-zero. This is the primary success metric.
  • vs starter eval win rate: may dip slightly because the policy trains less often against simple aggressive expansion, more often against lb-928's complex planning.
  • Value head calibration: should improve because V(s) now sees states from lb-928's distribution, which differs sharply from pure self-play.
  • Policy diversity: marginal; we're adding one extra opponent, not a league.

8.5 Risks

  • Learner games against lb-928 terminate fast (lb-928 wipes out weak policies in ~100 turns). Training samples per game drop accordingly, but 4 workers × 125 iters compensates.
  • If 50 % lb-928 is too strong, the learner might collapse into defensive play specialised against lb-928 rather than exploring diverse strategies. Easy to tune via --lb928-prob.
  • Advantage normalisation may behave oddly if batches contain a mix of "won easy vs self" and "got crushed by lb-928" — the mean/std will be dominated by whichever regime has more samples. We keep the advantage clip [−2, 2] to limit any single extreme sample.

9. Evaluation

9.1 Online during training

A separate eval_watcher.py process polls the saved checkpoint every 10 minutes. Whenever the .pt file's mtime changes, it loads the checkpoint, plays 10 games each vs starter and lb-928 (alternating seats), and appends a row to .impala_eval_watch.csv with the two win rates and wall time.

This runs on the spare GPU (GPU 1, with a hard 10 % memory cap) so it doesn't compete with the CPU-bound training workers.

9.2 Gate for submission

Before any new submission:

  • Win rate vs lb-928 ≥ 5 % (currently 0 %; a little is big)
  • Win rate vs previous submission ≥ 55 % (N = 40 local games, alternating seats)

Otherwise we keep iterating — a regression submission wastes a precious slot and the Kaggle "validation episode failed" error we already tripped once.

10. Deployment (Mode A: policy only)

At inference we do not run MCTS or any tree search. Each Kaggle turn:

  1. Encode obs via the same _encode_obs used in training.
  2. Forward pass through OrbitAgent (3–5 ms on CPU).
  3. Apply the sun-blocker mask to target_logits.
  4. For each owned planet, argmax of target_logits:
    • If class 0 (pass): skip — no move from this planet.
    • Else: translate (tgt_class, bkt_class) → env move [src_id, atan2(…), num_ships].
  5. Return list of moves.

The main.py bundle uses lazy model loading (defers torch.load until the first agent() call) to keep import time under the 1 s actTimeout. The remaining 2 s of bankable overage absorbs the first-call cost.

12. Key differences from the literature

Paper / SystemSimilarityDifference in our setup
AlphaStar (DeepMind 2019) PFSP opponent pool, per-unit action factorisation, teacher KL No league exploiters; single-teacher (none in v1); CPU-only 4 actors vs 16k TPUs.
Lux AI S1 winner (Pressman 2021) IMPALA-ish, shaped-reward decay, teacher KL to BC, per-unit heads No actual IMPALA V-trace; synchronous; no UPGO; Set Transformer replaces 24-block ResNet.
AlphaZero (Silver 2017) Shared policy+value tower No MCTS at training or inference (for mode A). Tree search is too expensive per-turn and Kaggle RTS winners have never used it.
PPO (Schulman 2017) Advantage normalisation, clipped advantages No importance ratio clip (on-policy); simpler loss formulation.
MAPPO (Yu 2022) Simultaneous-move multi-agent; shared critic Our critic is the same encoder shared with policy, not a separate centralised critic.

13. Open questions / future work

  • Model size: 0.9 M may bottleneck. If impala_v1 plateaus well below lb-928, widen to 4 M (d_model = 256, 8 layers).
  • League training: add frozen snapshots to the opponent pool as self_u10, self_u20, … every N iters (AlphaStar-lite). We keep the hook (was there in earlier PPO attempts) but don't use it in v1 or v2.
  • Inference-time MCTS (Mode B): if policy-only impala plateaus, Sampled MuZero / Gumbel MuZero via LightZero could add tactical depth at the cost of ~500 ms per turn. Risk: Kaggle actTimeout violations.
  • 180° augmentation: the board has centre symmetry; mirroring planets through (50, 50) doubles usable training data with no correctness cost.
  • Per-seat opponent sampling in self-play: right now all non-learner seats in a game use one policy choice. We could sample independently per seat per step, but this breaks the "what game was this" assumption for reward shaping.
  • Curriculum on game length: start with games capped at 200 steps, lift to 500 once the value head is stable. Would make early iterations ~2× faster.
