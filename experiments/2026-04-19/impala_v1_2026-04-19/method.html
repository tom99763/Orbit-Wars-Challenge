Orbit Wars — Lux-S1-Flavoured Self-Play Training

Method note for the impala_v1_2026-04-19 run. Written 2026-04-19 during training.

1. The Problem

Kaggle's Orbit Wars Featured Simulation Competition is a 2-or-4 player RTS on a 100×100 continuous board with a sun at the centre. Each turn, every player issues a list of moves: [source_planet_id, angle_radians, num_ships]. Planets produce ships, rotate around the sun (inner) or stay static (outer). Fleets travel in straight lines; crossing the sun destroys them. Combat is resolved by the largest attacking force minus the second-largest. The winner is whoever has the most ships (on planets + in fleets) after 500 turns, or the last survivor.

Important structural facts:

  • Perfect information — every player sees planets and fleets of every other player.
  • Simultaneous moves — all players submit actions at once per turn; there is no turn order.
  • Variable action space — each turn you emit K moves, one per planet you own, where K changes as you lose/gain planets.
  • Deterministic given seed — env is fully reproducible.
  • Sparse terminal reward — ±1 at game end, 0 elsewhere.

2. Starting Point: bc_v2

We previously trained a behaviour-cloning (BC) policy on a mixed dataset of scraped top-10 Kaggle replays + locally-generated starter self-play games. The resulting agent bc_v2:

  • 0.9 M-parameter Set-Transformer (permutation-invariant encoder over variable planet and fleet sets).
  • Two policy heads: target_planet ∈ {pass, planet_0, …, planet_N} and ships_bucket ∈ {25 %, 50 %, 75 %, 100 %}.
  • Value head: a scalar V(s) estimate.
  • Local win rate: 95 % vs random, 20 % vs starter, 0 % vs the rules-based lb-928 planner.

This run bootstraps from bc_v2 and improves the policy via online self-play.

3. The Method — Lux S1 Winner, Adapted

We base the design on Pressman et al. (2021), the only deep-RL agent to win a Kaggle simulation competition (Lux AI Season 1). Their recipe:

  1. IMPALA distributed actor-learner architecture.
  2. Policy gradient with V-trace off-policy correction.
  3. UPGO (upper-bound policy gradient optimisation).
  4. TD-λ value targets.
  5. Teacher-KL against a frozen BC network for stability.
  6. Dense reward shaping for the first ~20 M steps, then sparse only.
  7. Test-time augmentation (180° board rotation).
  8. AlphaStar-style self-play league.

What we keep, adapt, and skip

ElementKeep/Adapt/SkipWhy
Per-unit/per-planet action factorisation✅ Keep Our action is a set of moves per owned planet; joint log-prob is the sum across owned planets, same as Lux.
Reward shaping✅ Adapted to Orbit Wars Planet-count / production-rate / sun-kill penalty / kill-bonus. Decays linearly to zero over the first 50 iters so terminal ±1 eventually dominates.
Value head✅ Keep, shared encoder V(s) trained on every seat's terminal reward (winners +1 and losers −1).
Fully-convolutional ResNet backbone❌ Skip Orbit Wars has no natural grid; we use a Set Transformer over planet and fleet entities instead.
Frozen BC teacher KL❌ Skip (coef = 0) BC is our starting point, not a ceiling. Anchoring to it would limit the policy from improving past BC quality.
V-trace off-policy correction❌ Skip Our rollouts are synchronous and on-policy — we train on data generated by the same weights (or nearly so). Importance-sampling ratio ≈ 1.
UPGO❌ Skip (for now) UPGO needs accurate Q estimates we don't have; we use advantage normalisation + clip as a cheap stabiliser.
TD-λ⚠️ Simplified to γ-discounted MC return Episodes are short (≤ 500 steps); MC return is a reasonable baseline.
IMPALA async actor-learner❌ Sync multiprocessing insteadFour workers generate trajectories concurrently; master updates once per iter. Simpler, no replay buffer, no lag.
180° TTA❌ Not yet Free 2× data via centre-symmetry; worth adding if we need more signal.

4. Observation and Action Encoding

Observation is a tuple (planets, fleets, globals):

  • planets [N, 14]: owner one-hot, normalised position, radius, log ships, production one-hot, is-static, is-comet.
  • fleets [F, 9]: owner, position, sin/cos(angle), log ships, from-planet index, estimated-time-to-impact.
  • globals [16]: step fraction, angular velocity, remaining-turn fraction, player-slot one-hot, is-4p flag, per-side planet and ship tallies, comet-window indicator, rotation-phase sin/cos.

Architecture: input MLP per entity type + type embedding → 4-layer Set-Transformer (d = 128, 4 heads) → per-planet action heads + global-token value head. ~ 0.9 M parameters; CPU forward ~ 3–5 ms per sample.

5. Training Loop (per iter)

for iter in 0 .. 125:
    broadcast current model weights → 4 workers

    each worker plays one full game:
        random n_players ∈ {2, 4}
        every seat uses the SAME current model (pure self-play)
        every step, every seat's obs is encoded → forward → sampled action
        env.step([actions[0], actions[1], ...])

    after all 4 games complete:
        for every seat in every game:
            compute per-step shaped reward
            attach terminal reward (±1) to every record of that seat
            Monte-Carlo discount-sum → shaped return G_shape_t
            final return = shape_weight · G_shape_t + terminal_reward

    gradient step(s) on combined all-seat batch:
        advantage = return − V(s);  normalised + clipped to [−2, 2]
        policy_loss = − E[advantage · log π(action | state)]
        value_loss  = ½ · E[(V(s) − return)²]
        entropy_bonus = H(π(·|s)) over the target-planet head
        loss = policy_loss + 0.5 · value_loss − 0.01 · entropy

    if iter % 10 == 0: save checkpoint

Reward shaping

r_shape_t = 0.10 · Δ(my_planet_count) + 0.02 · Δ(my_production_rate) − 0.10 · (my_ships_lost_to_sun) / 100 + 0.05 · (enemy_ships_destroyed) / 100

Shape weight decays from 1.0 at iter 0 → 0.0 by iter 50. After iter 50 the policy trains purely on the sparse ±1 terminal signal, the authentic objective.

Why all seats feed back

This is a key adaptation. Naïve BC-from-winners wastes 50 % of 2p data and 75 % of 4p data. Instead, every seat's (obs, action, terminal_reward) trains the policy and value networks:

  • Winners (terminal +1, positive advantage): −(+adv) · log π → minimise → push log π up → take this action more.
  • Losers (terminal −1, negative advantage): −(−adv) · log π → minimise → push log π down → take this action less.
  • Both contribute to value-loss with their own terminal reward as the target, training V(s) to reflect true game outcomes.

The gradient signs are automatic once advantages are computed correctly; we do not need to split winners from losers explicitly.

6. Why Not MCTS (Mode B)?

We explored an AlphaZero-flavoured run with Sampled MCTS earlier (alphazero_v1_2026-04-19, killed). Two obstacles:

  1. Per-iter cost: ≥ 10 simulations × 150 tree descents × 4 games = ~ 8 minutes per iter on CPU. A 500-game run would take ~ 16 hours.
  2. No RTS winner on Kaggle has ever used inference-time MCTS. Lux S1 (Pressman 2021), Kore, Halite, Lux S2 winners all deployed policy-only agents. Simultaneous moves + huge variable action spaces make shallow MCTS noisy rather than helpful.

We keep the option open: if this run plateaus below lb-928, we can add tree search at training time as a policy-target refiner (mode A: MCTS for data gen, policy-only at inference).

7. Expected Behaviour

PhaseItersWhat should happen
Early0–20 Shape weight ~1.0; value loss drops fast (V(s) aligns with shaped returns); policy starts departing from BC prior; entropy rises as advantages push the policy to explore other actions.
Mid20–60 Shape weight decays; terminal reward takes over; win-rate vs starter should climb past BC's 20 % baseline.
Late60–125 Pure terminal reward; policy consolidates on strategies that actually win. Value loss should plateau; win rate vs starter targets ≥ 60 %, vs lb-928 any non-zero is progress.

8. What Could Still Go Wrong

  • Policy collapse — advantages dominated by noise; entropy drops to 0; agent picks one fixed action. Mitigation: entropy coef 0.01 + advantage normalisation + clip.
  • Value head chasing shape tail — when shape_weight → 0, V(s) suddenly has different target scale. Mitigation: shape decay is smooth (linear 50 iters), and terminal reward is always present in the return formula.
  • 0.9 M params too small for the diversity of policies self-play generates. If we plateau well below lb-928, widen to 4 M.
  • Pass-bias inheritance from bc_v2 — agent only emits one action per turn. Advantage-weighted updates should eventually correct this, but not guaranteed.

9. Deployment (Mode A)

After training we will:

  1. Copy final training/checkpoints/impala_v1.pt → submission bundle at submission/az_v1/model.pt.
  2. Keep the existing lazy-load main.py (defers torch.load off the first-tick critical path since Kaggle's remainingOverageTime = 2 s absorbs a slow first step).
  3. Tar + kaggle competitions submit orbit-wars -f submission.tar.gz.
  4. Poll competitions.EpisodeService via scrape_submission.py to track real-world ELO.

At inference we do not run MCTS. One forward pass per turn, pick argmax per owned planet, translate back to [src, angle, ships] tuples.
